INT 03 C - CPU-generated - BREAKPOINT
Desc:	generated by the one-byte breakpoint instruction (opcode CCh)
Notes:	used by debuggers to implement breakpoints, such as MS-DOS DEBUG's G
	  command
	also used by Turbo Pascal versions 1,2,3 when {$U+} specified
	return address points to byte following the breakpoint instruction
	called by Novell DOS 7 EMM386 after displaying an exception error
	  message (GPF, illegal opcodes, etc.) and before waiting for user
	  input
	called by QEMM-386 if the user presses 'I' (undocumented) when prompted
	  after an exception error message
	usually initialised to point at an IRET by DOS
	lDOS (FreeDOS, Enhanced DR-DOS, lMS-DOS) may call this for debugger
	  detection. this usually initialises CF to be clear. if the debugger
	  modifies it so CF is set (lDebug: r f CY) then the debugger is
	  detected and its INT 01 and INT 03 vectors are preserved.
	may be uninitialised by the ROM-BIOS when passing control to the next
	  boot loader. lDOS debugger detection has an option to run INT 03 only
	  if the vector appears valid (offset != FFFFh and segment != 0000h).
	  (not supported by the FreeDOS debugger detection.)
	see notes on INT 01"CPU" for the MS-DOS v7 debugger detection, which
	  also affects the INT 03 vector
BUGS:	MS-DOS Debug from the 2018 release of MS-DOS v2 fails to restore the
	  prior handler after setting it, even upon running the Q command.
	  at least as of MS-DOS v4.00 this was fixed.
SeeAlso: INT 01"CPU"
